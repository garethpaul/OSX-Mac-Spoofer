# Pre-Change Hardware Address Capture

status: planned

## Context

The live path reads the hardware address only after all mutation commands and
post-change verification succeed. If that final read fails, the interface has
already changed even though the CLI exits with an error, creating an ambiguous
partial-success result and delaying capture of restoration data until after
state mutation.

## Requirements

- Read and validate the current and hardware addresses before any mutation
  command executes.
- Abort without executing mutation commands when either pre-change lookup
  fails.
- Preserve the existing mutation command order, post-change equality check,
  successful output, dry-run isolation, validation, and timeout behavior.
- Add a mocked regression test and an ordering-sensitive static contract.
- Keep all verification non-privileged and free of generated bytecode.

## Scope Boundaries

- Do not execute real macOS networking commands.
- Do not add rollback automation, persistence, interface enumeration, new
  dependencies, or weaker address validation.
- Do not change the startup wrapper, workflow matrix, plist, or restoration
  procedure.

## Verification Plan

- Run the focused mocked tests and every standard Make gate from the repository
  root and an external working directory.
- Run hostile mutations against the lookup ordering, regression test, static
  contract, and completed plan evidence.
- Audit the intended diff, secrets, captured identifiers, and generated
  artifacts before committing.
